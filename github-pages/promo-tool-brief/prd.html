Retail Promotions Management Tool

Detailed product requirements document built from Nathan’s voice notes, 12 March 2026

This document is the detailed build scope for Aden. It captures the supplier workflow, buyer workflow, slot logic, weighting rules, pricing logic, margin engine, benchmark pricing integration, notification cadence, promo-period release logic, and downstream Shopfront execution requirements.

Product objective

Build an integrated supplier promotions management application inside the Paramount supplier portal that allows suppliers to submit promotion offers against a structured retail promo board, while enforcing commercial rules before buyer review and pushing approved promotions into Shopfront.

The system should improve promo quality, reduce manual back and forth, protect margin, create fair slot allocation, and make promo planning more scalable and disciplined.

Product philosophy

This is not just a form. It should behave like a guided commercial workflow engine. Before a buyer even touches a submission, the system should already be constraining suppliers to valid slots, valid products, valid pricing logic, valid slot allocation, and valid commercial thresholds.

Users and roles

Supplier user

  • enters via existing supplier portal
  • requires retail-specific permission/status
  • views open and historical promo periods
  • views available slots and prior approved pricing
  • submits promotions and materials

Buyer / Admin

  • buyers review, approve, or decline submissions
  • admins configure periods, slots, weighting, and margin rules
  • buyers receive daily digests
  • admins release promo periods in batches

Platform and access model

  • Build in Adobe App Builder
  • Use Adobe API Mesh
  • Suppliers log in through the existing Paramount website
  • Only users with the right retail permission/status see the promotions entry point
  • Clicking into the app should use SSO / OAuth for seamless handoff

Promo period model

Promotions operate on a fortnightly cycle. Promo periods must auto-generate using the week number in which the promotion starts.

  • P1-26
  • P3-26
  • P5-26
  • P7-26

Periods must support open and close windows, release batches, statuses, and historical viewing.

Category and slot structure

The app should use a visual promotional slotting board. Top-level categories include Spirits, RTD, Beer, Wine, Non-alcoholic, and Other. Each category supports configurable subcategories.

Examples include mainstream beer, international beer, craft beer, entry vodka, premium vodka, bourbon, scotch whisky, local gin, and international gin.

Each promo period needs configurable slot counts by category and subcategory.

Supplier dashboard behaviour

  • visual slot board, not just a list
  • toggle back to historical periods
  • see approved products and prices in filled slots
  • see supplier-specific slot availability
  • see unavailable / greyed-out slots where no valid product exists

Historical price visibility is intentional. It creates commercial transparency and pushes suppliers to bring stronger offers next time.

Product eligibility rules

  • Only core range, tier one, and tier two products may be promoted
  • Products must map to valid category/subcategory combinations
  • Suppliers only see their own eligible products
  • A product cannot run in two consecutive promo periods

Supplier weighting and slot allocation

Each supplier should have category-level slot entitlements based on market share or agreed weighting. This prevents a major supplier from flooding the board.

Example: Lion has 30% share in mainstream beer. There are 6 slots. Lion gets 2 of 6 slots. The UI should show “You have filled 0 of 2 slots”.

If the supplier exceeds allocation, block the submission and tell them to speak to the buyer.

Submission wizard

  1. Select category / slot
  2. Select eligible product
  3. Select one pack format only, case, 6-pack, or single
  4. Review wholesale cost and supply discount
  5. Enter scan deal
  6. Enter proposed promotional retail price
  7. Review margin and benchmark prices
  8. Upload promotional assets
  9. Submit

Commercial pricing logic

The pricing screen should show:

  • Paramount wholesale cost
  • supply discount
  • net retail buy price
  • scan deal input
  • proposed promo retail price
  • dollar margin
  • percentage margin

Calculations must update in real time. If the promo sits below threshold, margin should show in red and submission should be blocked.

Margin rule engine

Margin thresholds must be configurable by supplier, category, and pack format. Nathan explicitly wants the flexibility to accept different minimum margins for different suppliers.

Suggested precedence:

  1. supplier + category + pack override
  2. supplier + category override
  3. category + pack default
  4. category default

Market benchmark pricing

Integrate the existing price scraping tool and show:

  • lowest observed market price in the last 4 weeks
  • average / most common market price in the last 4 weeks
  • retailer name beside each
  • date recorded beside each

This is a deliberate behavioural tool. Suppliers should see the real market context while entering their offer, so weak pricing becomes harder to justify.

Buyer workflow

  • buyers receive daily summary emails of new submissions
  • buyers log into a queue filtered by category and period
  • buyers review pricing, margin, benchmark context, and assets
  • buyers approve or decline
  • decline requires a reason code and optional notes

Shopfront integration

Approval should trigger the downstream Shopfront load. That push should include SKU, pack format, scan deal, promo price, and promo dates.

If the Shopfront push fails, the system should capture the failure, flag it internally, and allow retry.

Notifications and release workflow

  • notify suppliers when a new period or batch of periods opens
  • support release of multiple future promo periods in one batch
  • show countdown timer for supplier submission window
  • notify suppliers end of day with approved and declined outcomes
  • notify buyers daily with queue digest

Admin modules

  • period management
  • slot builder
  • category and subcategory config
  • supplier eligibility config
  • product mapping config
  • margin rule config
  • supplier weighting config
  • reason code config
  • release management
  • notification templates

MVP scope

  • supplier permission gating

  • margin validation and benchmark pricing display

  • decline reasons

Acceptance criteria, highlights

  • Users without the required retail permission cannot access the app.
  • Users with permission can enter without separate login.
  • Suppliers only see eligible products and valid slots.
  • Same product cannot run in consecutive periods.
  • Supplier slot caps are enforced.
  • Below-threshold promos show red validation state and cannot submit.
  • Buyer decline requires a reason code.
  • Buyer approval triggers Shopfront integration.

Final note for Aden

The key thing here is preserving the commercial logic, not just the screens. If the app gets the slot allocation, product eligibility, margin validation, benchmark pricing, and approval workflow right, it will do a big chunk of the buyer’s job before the buyer ever opens the queue.
